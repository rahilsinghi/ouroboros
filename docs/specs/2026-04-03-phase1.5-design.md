# Phase 1.5: Fix Feedback Loop + Smarter Agents

## Goal

Make the Ouroboros self-improvement loop produce at least one successful merge when targeting its own codebase. Then make the agents smarter so they produce higher-quality changes.

## Context

Phase 1 proved the loop runs end-to-end: Observer identifies weaknesses, Strategist proposes changes, Implementer writes code in a worktree, Evaluator decides merge/rollback. But after 2 real iterations, nothing has been merged. The root causes are scoring miscalibration and agent naivety.

### Current Scores (from live run 2026-04-02)
| Dimension | Score | Why |
|-----------|-------|-----|
| code_quality | 0.24 | ruff finds real violations in ouroboros code |
| correctness | 0.00 | test runner points pytest at wrong dir |
| efficiency | 1.00 | token count vs baseline (trivially met) |
| regression | 1.00 | no history, so nothing to regress |
| tool_selection | 1.00 | placeholder |
| real_world | 0.50 | placeholder |

### Why Nothing Merges
1. **Merge gate requires improvement** — at least one dimension must improve beyond 0.02 noise tolerance
2. **Correctness is 0.0 before AND after** — no improvement possible without fixing the test path
3. **Code quality drops** when agents make changes (they introduce violations)
4. **Agents target blocked files** — waste iterations proposing changes to runner.py, config.py, etc.

## Part A: Fix the Feedback Loop

### A1. Fix Correctness Scoring

**Problem:** `_run_tests()` in `runner.py` runs `pytest` against the target dir. When `target_path = ouroboros/`, it looks for tests inside the package itself. But tests live in `tests/ouroboros/`.

**Fix:** The `test_command` in config already specifies `python -m pytest tests/ouroboros/ -v`. But `_run_tests()` ignores this and hardcodes `pytest <target_path>/..`. Change `_run_tests()` to use the configured `test_command` and run it from the repo root (or worktree root), not from the target dir.

**Acceptance criteria:**
- `_run_tests()` uses the configured test command string
- Running `run_scoreboard()` against the repo root produces `correctness > 0.0`
- Tests for the scorer still pass

### A2. Improve Test Result Parsing

**Problem:** `_run_tests()` parses pytest `-v` output looking for `PASSED`/`FAILED` strings. This is fragile — pytest's output format varies by version and configuration. Also, the word count for "token_count" is meaningless.

**Fix:**
- Use `--tb=no -q` flags for cleaner output
- Parse the summary line (`X passed, Y failed`) as a fallback
- Replace word-count "tokens" with file-level character count of source files in `target_path` (actual proxy for code size)

### A3. Filter Blocked Files from Source Context

**Problem:** `_read_target_files()` reads ALL `.py` files under the target path and sends them to the Strategist. This includes blocked files like `config.py`, `loop.py`, `evaluator.py` — which the agents then try to modify.

**Fix:** Filter out files whose paths match `sandbox_blocked_paths` before sending them to the Strategist. If the agent can't see a file, it won't try to change it.

**Acceptance criteria:**
- `_read_target_files()` accepts a `blocked_paths` parameter
- Files matching any blocked path prefix are excluded from the dict
- Strategist only sees modifiable files

### A4. Improve JSON Robustness

**Problem:** LLM responses are sometimes truncated (hitting max_tokens) or wrapped in prose. The JSON repair in `parse_json()` is too naive — closing braces doesn't fix truncated string values containing code.

**Fix:**
- Increase `max_tokens` for Strategist and Implementer (4096 → 8192)
- Add a retry mechanism: if JSON parse fails, re-prompt the LLM with "Your previous response was invalid JSON. Please respond with ONLY the JSON object."
- Cap retry at 1 attempt

**Acceptance criteria:**
- Strategist and Implementer use 8192 max_tokens
- Failed JSON parse triggers one retry
- Retry prompt includes the parse error message

### A5. Add Cost Tracking

**Problem:** Budget config exists (`max_usd_per_run: 10.0`) but isn't enforced. No visibility into spend.

**Fix:**
- Track input/output tokens per agent call in `BaseAgent.call()`
- Add `tokens_to_usd()` using published Claude pricing
- Check cumulative cost in `loop.py` before each iteration
- Log cost per iteration to ledger
- Print cost summary after loop completes

**Pricing (Sonnet 4.6):** $3/M input, $15/M output

## Part C: Smarter Agents

### C1. Strategist: Provide Richer Context

**Problem:** The Strategist gets source files + observation + ledger summary. But it doesn't know:
- What tests exist and what they test
- What ruff violations exist specifically
- What the code quality score breakdown is

**Fix:** Include in the Strategist prompt:
- Ruff violation details (file, line, rule, message)
- Radon complexity details per function
- Test names and their pass/fail status
- The specific blocked paths list (already done, but reinforce)

### C2. Implementer: Validate Before Committing

**Problem:** The Implementer writes code and commits without checking if it's valid. The Evaluator only checks scoreboard scores, not syntax or type errors.

**Fix:** After writing files but before committing:
1. Run `ruff check` on modified files
2. Run `python -c "import ast; ast.parse(open('file').read())"` for syntax check
3. If either fails, attempt one self-correction: send the error back to the LLM
4. Only commit if validation passes

### C3. Observer: Use Structured Dimension Analysis

**Problem:** Observer just gets score numbers and traces. It lacks context about what each dimension measures and how to improve it.

**Fix:** Include dimension-specific guidance in the Observer prompt:
- `code_quality < 0.5` → "Run ruff check to see specific violations"
- `correctness < 1.0` → "Examine which tests fail and what they assert"
- Each dimension gets a brief "how to investigate" section

### C4. Evaluator: Richer Merge Criteria

**Problem:** Merge gate is purely numeric — it can't distinguish between a meaningful improvement and a lucky one (e.g., removing a test to boost correctness).

**Fix:** Add a "sanity check" step in the Evaluator:
- If correctness improved, verify that no tests were deleted
- If code_quality improved, verify the diff doesn't just delete code
- Check that the diff is non-trivial (> 5 lines changed)
- Log the reasoning

### C5. Multi-Iteration Learning

**Problem:** Each iteration starts fresh — the Strategist doesn't learn from failed attempts beyond the ledger summary.

**Fix:** Expand the ledger summary to include:
- Specific failure reasons (not just "merge gate failed")
- Which files were attempted and what happened
- A "do not repeat" section listing hypotheses that failed
- The Strategist prompt explicitly says "avoid these failed approaches"

- [ ] `correctness > 0.0` on baseline scoreboard
- [ ] At least one iteration produces a successful merge
- [ ] No iterations crash due to JSON parsing errors
- [ ] Cost is tracked and displayed per iteration

- [ ] Strategist receives ruff/radon details, not just scores
- [ ] Failed hypotheses are explicitly listed in strategist context
- [ ] At least 2 out of 5 iterations produce a successful merge
